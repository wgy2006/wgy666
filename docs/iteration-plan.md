## 2. 本迭代目标

本迭代不以完整编码实现为主要目标，而是完成项目启动阶段的需求、设计和原型工作。具体目标如下：

1. 明确项目定位、用户群体、核心价值和系统边界，形成《Vision文档》。
2. 分析并定义需求，建立Usecase模型，包括Usecase图和主要Usecase规约。
3. 初步确定后端、前端、数据库、向量检索、大模型调用、多Agent协同和GitHub API接入方案。
4. 完成Web端界面原型，覆盖仓库接入、仓库问答、Issue分析、Issue处理建议等核心页面。
5. 组织需求文档和界面原型评审，记录问题并完成一轮修改。
6. 编写《迭代评估报告》，总结本迭代完成情况、问题、风险和下一步计划。
7. 使用Git管理文档、模型、原型和后续代码，形成基本的分支和提交规范。

### 3.2 本迭代暂不包含

1. 完整的后端业务实现。
2. 完整的GitHub Webhook监听和自动PR创建功能。
3. 真实大模型Agent稳定运行。
4. 多项目高并发性能压测。
5. 长期记忆和FAQ自动生成的完整实现。

这些内容放入后续迭代，根据本迭代评审结果继续细化。

## 6. 需求优先级

采用MoSCoW方法划分优先级。

| 优先级 | 需求 | 说明 |
| --- | --- | --- |
| Must | GitHub仓库接入 | 系统基础能力，后续问答和Issue分析都依赖仓库数据 |
| Must | 仓库知识库构建 | 包括代码结构、README、Issue、PR、提交记录等信息整理 |
| Must | 仓库智能问答 | 支持What、Where、How类型问题 |
| Must | Issue智能分析 | 识别重复、咨询、信息不足、缺陷、功能建议等类型 |
| Must | Web浏览器客户端 | 满足非功能性需求 |
| Should | 自动生成Issue回复内容和理由 | 对无需改代码的问题提高维护效率 |
| Should | 响应式界面 | 适应不同分辨率和尺寸 |
| Should | 多项目并发处理设计 | 本迭代给出架构方案，后续实现 |
| Could | Issue自动修复与PR创建 | 进阶功能，视后续时间实现核心流程 |
| Could | 长期记忆和维护者偏好学习 | 进阶功能，可先设计数据结构 |
| Could | 高频问题FAQ生成 | 可以作为Issue分析后的扩展能力 |

## 8. 技术选型初案

| 模块 | 候选技术 | 初步选择 | 理由 |
| --- | --- | --- | --- |
| 前端 | React、Vue、Next.js | React + Vite 或 Next.js | 生态成熟，适合快速实现Web原型和响应式界面 |
| 后端 | Python FastAPI、Node.js NestJS、Spring Boot | Python FastAPI | 便于集成LLM、RAG、向量库和AI工具链 |
| GitHub接入 | GitHub REST API、GraphQL API、Webhook | REST API + Webhook，必要时补充GraphQL | REST API易上手，Webhook适合监听Issue事件 |
| 数据库 | PostgreSQL、MySQL、SQLite | PostgreSQL | 适合存储仓库、Issue、PR、任务和长期记忆 |
| 向量库 | pgvector、Chroma、Milvus | pgvector 或 Chroma | 课程项目中部署成本较低，便于RAG检索 |
| RAG框架 | LangChain、LlamaIndex、自研轻量流程 | LlamaIndex 或轻量自研 | 支持文档切分、索引和检索，适合仓库知识库 |
| Agent协同 | LangGraph、AutoGen、自研任务流 | LangGraph 或自研任务流 | 方便表达Issue分析、定位、修复、PR等多步骤流程 |
| 原型工具 | Figma、墨刀、HTML/CSS | Figma 或前端静态页面 | 便于评审和展示 |
| 版本管理 | Git + GitHub | Git + GitHub | 满足课程要求，支持协作和历史追踪 |


建议分支规范：

| 分支 | 用途 |
| --- | --- |
| main | 稳定版本，只合并评审通过的内容 |
| docs/iteration-1 | 第1次迭代文档工作 |
| prototype/ui-v1 | 界面原型工作 |
| feature/github-sync | 后续GitHub仓库接入功能 |
| feature/issue-analysis | 后续Issue智能分析功能 |

建议提交信息格式：

```text
docs: add iteration plan
docs: update vision document
design: add usecase diagram
prototype: add repository qa page
```

基本协作流程：

1. 每名成员从自己的任务分支提交。
2. 提交前检查文档格式、图片路径和文件命名。
3. 通过Pull Request合并到main分支。
4. PR中说明本次修改内容、关联任务和需要评审的问题。
5. 评审后再合并，避免直接向main推送未检查内容。

## 11. 风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
| --- | --- | --- | --- |
| GitHub API、Webhook或认证机制学习成本较高 | 影响后续仓库接入 | 中 | 本迭代先完成API调研和最小验证，后续先做公开仓库读取 |
| RAG和多Agent技术复杂 | 影响智能问答和Issue分析质量 | 高 | 先实现简单检索增强问答，再逐步加入Agent任务拆分 |
| 需求范围过大 | 影响迭代完成度 | 高 | 优先完成基本需求，进阶需求按时间裁剪 |
| 小组成员技术栈不统一 | 影响协作效率 | 中 | 统一前后端框架和代码规范，安排学习记录和示例工程 |
| 文档和原型不一致 | 影响评审质量 | 中 | 由配置管理员统一整合，每次评审前检查用例、页面和需求映射 |
| Git协作冲突 | 影响提交进度 | 中 | 使用分支开发和PR评审，避免多人同时修改同一文件 |

1. 《迭代计划》已提交到GitHub仓库。
